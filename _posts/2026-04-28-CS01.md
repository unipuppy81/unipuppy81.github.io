# 메모리 구조와 GC 이해하기

프로그램이 데이터를 어떻게 메모리에 저장하고 사용하는지를 알아보자.  
메모리를 어떻게 사용하는지 이해하면 성능 문제, GC Alloc, 프레임 드랍, 메모리 누수를 분석하는데 도움이 된다.

---

## 1. 프로세스 메모리 구조

프로그램이 실행되면 운영체제는 해당 프로그램에 메모리 공간을 할당한다.  
메모리 공간은 일반적으로 아래와 같다.

![프로세스 메모리 구조](/assets/images/memory/memory-layout.png)

| 영역 | 설명 |
| --- | --- |
| Code 영역 | 실행할 기계어 코드가 저장되는 영역 |
| Data 영역 | 전역 변수, static 변수가 저장되는 영역 |
| Heap 영역 | 런타임 중 동적으로 할당되는 메모리 영역 |
| Stack 영역 | 함수 호출, 지역 변수, 매개변수 등이 저장되는 영역 |

게임 개발 시 주의깊게 봐야하는 영역은 **Heap**과 **Stack**이다.

---

## 2. Stack

Stack은 함수 호출과 함께 자동으로 관리되는 메모리 영역이다.

대표적으로 int, float, bool, struct, enum 과 같은 값 타입이 이곳에 저장된다.

```csharp
void Test()
{
    int hp = 100;
    float speed = 5.0f;
}
```

위 코드에서 `hp`, `speed` 같은 지역 변수는 일반적으로 Stack에 저장된다.  
함수가 끝나면 해당 함수 호출에 사용된 Stack Frame은 제거된다.

Stack의 특징은 다음과 같다.

- 할당과 해제가 빠르다.
- 함수 호출 흐름에 따라 자동으로 정리된다.
- 크기가 제한적이다.
- 지역 변수와 함수 매개변수 등이 주로 저장된다.

다만 C#에서는 값 타입이라고 해서 항상 Stack에만 저장되는 것은 아니다.  
중요한 기준은 **타입 자체보다 그 데이터가 어디에 포함되어 있는가**이다.

```csharp
class Player
{
    public int hp;
}
```

`hp`는 `int`이므로 값 타입이지만, `Player` 객체의 필드다.  
`Player` 객체가 Heap에 생성되면 `hp`도 객체 내부에 포함되어 Heap에 존재한다.

반대로 `struct` 안에 `class` 타입 필드가 있을 수도 있다.

```csharp
class Weapon
{
    public int damage;
}

struct PlayerData
{
    public int hp;
    public Weapon weapon;
}
```

이 경우 `PlayerData`가 Stack에 생성되더라도 `weapon` 객체 자체가 Stack에 들어가는 것은 아니다.  
`PlayerData` 내부에는 `Weapon` 객체를 가리키는 참조값만 저장되고, `new Weapon()`으로 생성된 실제 객체는 Heap에 저장된다.

```text
Stack
└─ PlayerData
   ├─ hp = 100
   └─ weapon = Heap의 Weapon 객체 참조값

Heap
└─ Weapon 객체
   └─ damage = 10
```

정리하면 다음과 같다.

| 상황 | 저장 위치 |
| --- | --- |
| 함수 내부의 값 타입 지역 변수 | 일반적으로 Stack |
| 참조 타입 객체 | Heap |
| class 내부의 값 타입 필드 | 객체 내부에 포함 |
| struct 내부의 class 필드 | 참조값만 struct 내부에 포함, 실제 객체는 Heap |

즉, Stack을 이해할 때는 “값 타입은 무조건 Stack”이라고 외우기보다,  
**값 자체가 어디에 포함되는지와 참조 타입 객체가 어디에 생성되는지**를 구분해서 보는 것이 좋다.

---

## 3. Heap

Heap은 런타임 중 동적으로 객체를 생성할 때 사용하는 메모리 영역이다.

대표적으로 class, string, array, delegate 과 같은  참조 타입 객체가 Heap에 저장된다.

```csharp
Player player = new Player();
```

`new Player()`로 생성된 객체는 Heap에 할당된다.  
변수 `player`는 해당 객체를 가리키는 참조를 가진다.

Heap의 특징은 다음과 같다.

- 객체의 생명주기가 함수 호출보다 길 수 있다.
- 참조가 남아 있으면 계속 메모리에 유지된다.
- 사용하지 않는 객체는 GC가 수거한다.
- 할당과 해제가 Stack보다 상대적으로 비싸다.

Unity에서 `new`, 문자열 조합, LINQ, 박싱, 클로저, 일부 foreach 사용 등은 Heap 할당을 만들 수 있다.  
이런 할당이 매 프레임 반복되면 GC가 자주 발생하고 프레임 드랍으로 이어질 수 있다.

---

## 4. 값 타입과 참조 타입

C#에서는 타입을 크게 값 타입과 참조 타입으로 나눌 수 있다.

| 구분 | 예시 | 특징 |
| --- | --- | --- |
| 값 타입 | int, float, bool, struct, enum | 값 자체를 저장 |
| 참조 타입 | class, string, array, delegate | 객체의 참조를 저장 |

예를 들어 `struct`는 값 타입이다.

```csharp
struct Position
{
    public float x;
    public float y;
}
```

값 타입은 복사 시 값 자체가 복사된다.

```csharp
Position a = new Position();
Position b = a;
```

이 경우 `b`는 `a`와 별개의 값을 가진다.

반면 클래스는 참조 타입이다.

```csharp
Player a = new Player();
Player b = a;
```

이 경우 `a`와 `b`는 같은 객체를 가리킨다.  
따라서 `b.hp`를 변경하면 `a.hp`도 변경된 것처럼 보인다.

## 5. Garbage Collector

### 5-1. GC가 필요한 이유

C++은 개발자가 메모리를 직접 관리하지만,
C#은 개발자가 직접 메모리를 해제하지 않는다.  

대신 Garbage Collector, 즉 GC가 더 이상 사용되지 않는 객체를 찾아 메모리를 회수한다.

```csharp
void CreateEnemy()
{
    Enemy enemy = new Enemy();
}
```

함수가 끝나면 지역 변수 `enemy`는 사라지지만 `new Enemy()`로 생성된 객체는 Heap에 남게 된다.

이 객체는 더 이상 참조하는 곳이 없다면 GC의 수거 대상이 된다.

이러한 동작 흐름을 마크 앤 스윕 방식이라고 부른다.

Mark : 현재 접근 가능한 객체를 탐색하고 표시한다.  
Sweep : 표시되지 않은 객체, 즉 더 이상 참조되지 않는 객체의 메모리를 회수한다.

즉, GC는 **참조되지 않는 객체를 자동으로 정리하는 시스템**이다.


### 5.2 GC Alloc이 문제가 되는 이유

GC 자체가 항상 문제는 아니다.  
문제는 **짧은 시간에 많은 Heap 할당이 반복되는 경우**다.

게임은 보통 1초에 30~60프레임 이상 실행된다.  
매 프레임마다 불필요한 객체가 생성되면 Heap에 쓰레기 객체가 계속 쌓인다.

```csharp
void Update()
{
    string text = "HP : " + hp.ToString();
}
```

위 코드는 매 프레임 문자열 객체를 새로 만들 수 있다.  
이런 할당이 누적되면 GC가 더 자주 실행된다.

GC가 실행되는 동안 메인 스레드가 멈추거나 부하가 증가할 수 있다.  
Unity 게임에서는 이 순간 프레임이 튀거나 끊김이 발생할 수 있다.

## 6. Unity에서 자주 발생하는 GC Alloc

Unity에서 자주 볼 수 있는 GC Alloc 원인은 다음과 같다.

### 6-1. Update 내부의 new

```csharp
void Update()
{
    Vector3[] points = new Vector3[10];
}
```

매 프레임 배열을 새로 생성하면 Heap 할당이 반복된다.

개선 방향은 다음과 같다.

```csharp
private Vector3[] points = new Vector3[10];

void Update()
{
    // 기존 배열 재사용
}
```

---

### 6-2. 문자열 조합

C#에서 문자열(string) 은 불변성을 가진다.
이는 문자열 수정 시 기존 string 객체에 수정을 하는 것이 아닌
새로운 string 객체가 생성되고, 기존 string 객체는 더 이상 참조되지 않으면 GC 수거 대상이 된다.

따라서 많은 string 연산이 필요한 경우 StringBuilder 사용이 권장된다.

---

### 6-3. LINQ 사용

```csharp
var aliveEnemies = enemies.Where(e => e.IsAlive).ToList();
```

LINQ는 코드가 간결하지만 내부적으로 할당이 발생할 수 있다.  
성능이 중요한 구간에서는 명시적인 반복문이 더 적합할 수 있다.

```csharp
aliveEnemies.Clear();

for (int i = 0; i < enemies.Count; i++)
{
    if (enemies[i].IsAlive)
        aliveEnemies.Add(enemies[i]);
}
```

---

### 6-4. 박싱

박싱은 값 타입을 object나 인터페이스 타입으로 다룰 때 발생할 수 있다.

```csharp
int value = 10;
object boxed = value;
```

`int`는 값 타입이지만 `object`에 담기면서 Heap에 객체가 생성된다.

로그 출력, 인터페이스 호출, 컬렉션 사용 중 의도하지 않은 박싱이 발생할 수 있으므로 Profiler로 확인하는 것이 좋다.

제네릭 컬렉션을 사용하면 값 타입을 object로 다루는 상황을 줄여 박싱을 피할 수 있다.

---

## 7. GC를 줄이는 기본 방향

GC Alloc을 줄이는 핵심은 단순하다.

> 반복 실행되는 구간에서 불필요한 Heap 할당을 줄이는 것

실무적으로는 다음 기준을 적용할 수 있다.

| 상황 | 개선 방향 |
| --- | --- |
| 매 프레임 new 사용 | 미리 생성 후 재사용 |
| 자주 생성되는 오브젝트 | Object Pool 사용 |
| 문자열 반복 생성 | 변경 시점에만 갱신 |
| LINQ 사용 | 성능 구간에서는 for문 사용 |
| 임시 List 생성 | Clear 후 재사용 |
| Instantiate/Destroy 반복 | Pooling 적용 |

특히 Unity에서는 `Update`, `LateUpdate`, `FixedUpdate`, 반복 호출되는 UI 갱신 함수에서 GC Alloc을 우선 확인하는 것이 좋다.
